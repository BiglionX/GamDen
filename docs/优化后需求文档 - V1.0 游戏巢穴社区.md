## 1. 核心功能与边界定义

### 1.1 产品定位
**一句话定位**：GamDen是一个不依赖推荐算法的游戏社交平台——玩家通过邀请码入驻获得专属领地，在AI守护灵（Agent）引导下建立游戏社交圈，通过邀请好友扩张领地版图。

**核心价值**：
- 反算法：内容展示按时间线/地理位置，不依赖推荐算法
- 领地感：每个玩家拥有唯一坐标和可视化领地
- 社交链：邀请关系决定领地相邻关系，形成聚居区

### 2.3 权限校验规则
1. **API层校验**：所有API请求必须在Header中携带JWT Token，服务端校验用户角色和权限
2. **前端展示校验**：根据用户角色隐藏无权限的按钮和入口
3. **数据权限**：普通玩家只能查看自己的数据，运营人员可查看所有数据
4. **操作权限**：删除/冻结等敏感操作必须记录操作日志（operator_id, action, target, timestamp）

## 3. 关键业务流程梳理

### 3.1 用户注册与入驻流程

```
[营销H5] 用户访问 → 填写设备+常用游戏 → 提交申请
                                              ↓
                                    [后台] 运营审核（手动/自动）
                                              ↓
                                    [后台] 生成邀请码 → 短信/邮件通知用户
                                              ↓
[App] 用户下载App → 输入邀请码 → 选择守护灵 → 注册成功
                                              ↓
                                    [系统] 分配随机领地坐标（保证与邀请人在±10格内）
                                              ↓
                                    [系统] 初始化领地Lv.1、金币（初始100）
                                              ↓
                                    [Agent] 发送欢迎语（按守护灵类型）
```

**关键规则**：
- 邀请码有效期：30天，过期需重新申请
- 坐标分配算法：优先分配在邀请人周围±10格范围内，如无空位则随机分配
- 坐标冲突处理：如随机坐标已被占用，自动向右偏移+1 until找到空位

### 3.2 领地升级流程

```
[用户] 执行动作（签到/发帖/邀请好友/防御野兽潮）
                                              ↓
                                    [系统] 计算经验值（经验值规则见下表）
                                              ↓
                                    经验值达到阈值？
                                              ↓
                                    [系统] 升级领地 + 推送Agent通知
                                              ↓
                                    [用户] 查看新等级领地元素图
```

**经验值获取规则**：

| 行为 | 经验值 | 每日上限 | 备注 |
|------|--------|----------|------|
| 每日签到 | 10 | 10 | 连续签到额外+5/天（上限30） |
| 发帖 | 5 | 50 | 被删除帖子扣除相应经验 |
| 回帖 | 2 | 20 | - |
| 邀请好友入驻 | 50 | 无 | 好友需活跃7天以上才计入 |
| 防御野兽潮 | 20 | 200 | - |

**等级阈值**：

| 等级 | 所需经验 | 解锁条件 | 元素图（机械师/精灵/占星师） |
|------|----------|----------|------------------------------|
| Lv.1 | 0 | 注册即得 | ⚙️齿轮 / 🌱树苗 / 🔮星尘 |
| Lv.2 | 100 | - | 🔧铁砧 / 🌿小树林 / ✨星辉 |
| Lv.3 | 300 | 邀请1人 | ⛏️铁匠铺 / 🏡木屋 / 🔭观星台 |
| Lv.4 | 600 | 邀请3人 | 🔥锻炉 / 🏘️石屋群 / 🌌星门 |
| Lv.5 | 1000 | 活跃14天+邀请5人 | ⚒️铸造厂 / 🏯小型寨落 / 🌀星阵 |

### 3.3 邀请裂变与小程序的衔接流程

```
[用户A] 分享邀请码/链接 → [用户B] 输入邀请码注册成功
                                              ↓
                                    [系统] 记录邀请关系（invite_records表）
                                              ↓
                                    [用户A] 查看邀请进度（已邀请X人）
                                              ↓
                                    已邀请≥3人？
                                              ↓
                                    [系统] 自动生成个人小程序码和链接
                                              ↓
                                    [用户A] 分享小程序码到微信群/朋友圈
                                              ↓
                                    [新用户] 扫码 → 进入用户A的领地展示页
                                              ↓
                                    点击任意按钮？
                                              ↓
                                    [小程序] 弹窗"下载GamDen App，加入巢穴"
                                              ↓
                                    [新用户] 下载App → 输入用户A的邀请码注册
```

**关键规则**：
- 邀请进度条显示"已招募X/10名拓荒者"（目标10人，但解锁小程序仅需3人）
- 小程序页面内容：领地缩略图（当前等级）、邀请进度条、用户自定义签名（10字）、入驻天数、最新动态
- 小程序所有交互按钮统一引导回App，不在小程序内实现深度交互

### 3.4 内容发布与审核流程

```
[用户] 发布内容（签名/发帖/回帖）
                                              ↓
                                    [系统] 调用腾讯云内容安全API（AI初审）
                                              ↓
                                    AI审核通过？
                                   ↓                    ↓
                                是                      否
                                ↓                       ↓
                    [系统] 内容发布成功    [系统] 内容进入人工复审池
                                ↓                       ↓
                    [用户] 内容可见           [运营] 人工审核
                                                        ↓
                                              审核通过？          审核拒绝？
                                              ↓                   ↓
                                    [系统] 内容发布成功    [系统] 通知用户+记录违规
```

**审核规则**：
- 用户签名：≤20字符，AI初审+人工复审（如AI不确定）
- 俱乐部帖子：≤500字符，AI初审+人工复审
- 回帖：≤200字符，AI初审（高风险才进入人工复审）
- 敏感词库：政治敏感、色情、暴力、人身攻击、商业广告
- 违规处理梯度：第一次警告 → 第二次禁言24小时 → 第三次冻结账号

### 3.5 野兽潮防御流程

```
[系统] 定时任务（每小时检查一次）
                                              ↓
                                    检查无人区（无玩家区域）？
                                              ↓
                                    随机触发野兽潮（概率20%）
                                              ↓
                                    [系统] 生成野兽潮事件（等级1~5，对应威胁度）
                                              ↓
                                    附近有玩家领地（±20格）？
                                              ↓
                                    [系统] 推送威胁通知给附近玩家（Agent拟人化）
                                              ↓
                                    [用户] 查看野兽潮状态（/api/beast/status）
                                              ↓
                                    自动防御（无需手动操作）
                                              ↓
                                    防御成功（基于领地等级判定，Lv.3以上100%成功）
                                              ↓
                                    [系统] 发放金币奖励（50~200，基于野兽潮等级）
                                    [Agent] 发送防御成功通知
```

**关键规则**：
- 野兽潮刷新参数：时间（每小时）、频率（20%概率）、强度（1~5级），由运营在后台配置
- 防御失败处理：Lv.3以下领地有30%概率防御失败，失败后领地等级-1（最低Lv.1）
- V2.0才支持手动攻击和组队讨伐

### 3.6 金币系统流程

**金币获取途径**：

| 途径 | 数量 | 限制 | 备注 |
|------|------|------|------|
| 注册奖励 | 100 | 一次性 | - |
| 每日签到 | 10 | 每日1次 | 连续签到额外+5/天 |
| 发帖 | 5 | 每日上限50 | 被删除帖子扣除相应金币 |
| 回帖 | 2 | 每日上限20 | - |
| 邀请好友入驻 | 50 | 无 | 好友需活跃7天以上 |
| 防御野兽潮 | 50~200 | 无 | 基于野兽潮等级 |
| 每周活跃奖励 | 100 | 每周1次 | 连续7天登录 |

**金币消耗途径**：

| 途径 | 消耗 | 备注 |
|------|------|------|
| 兑换头像框 | 200 | 永久使用 |
| 兑换chat气泡 | 150 | 永久使用 |
| 兑换特殊签名（30字） | 300 | 30天有效期 |
| V2.0规划 | - | 实体周边、盲盒 |

**关键规则**：
- 金币与人民币不可兑换（V1.0）
- 金币余额上限：99999
- 金币不可转账/赠送

## 4. 非功能性需求

### 4.1 性能需求

| 指标 | 要求 | 备注 |
|------|------|------|
| **App启动时间** | ≤2秒（冷启动） | 包括初始化OpenIM |
| **页面加载时间** | ≤1秒 | 地图页≤2秒（需加载邻居数据） |
| **API响应时间** | ≤500ms（P95） | 复杂查询≤1秒 |
| **并发用户数** | 支持1000在线用户 | V1.0目标DAU 500 |
| **数据库连接池** | 最大100连接 | MySQL连接池配置 |
| **Redis缓存命中率** | ≥90% | 用户会话、领地信息缓存 |
| **图片加载** | WebP格式，≤200KB | 领地元素图、头像 |
| **H5打开速度** | ≤1秒 | 营销H5首屏渲染 |

### 4.2 安全性需求

| 类别 | 需求 | 实现方案 |
|------|------|----------|
| **身份认证** | JWT Token，有效期7天，支持刷新 | Token存储在SecureStore（App）、HttpOnly Cookie（Web） |
| **密码存储** | bcrypt加密，salt rounds≥10 | 不可明文存储 |
| **API防护** | 防CSRF、XSS、SQL注入 | ORM参数化查询、输入验证 |
| **敏感数据** | 手机号、邀请码脱敏展示 | 手机号显示138****1234 |
| **内容安全** | 所有UGC必须经过AI内容安全API | 腾讯云/阿里云内容安全API |
| **邀请码防刷** | 邀请码一次性使用，绑定设备ID | 同一设备不可重复注册 |
| **限流** | API限流：100次/分钟/用户 | Redis计数 |
| **日志审计** | 所有敏感操作记录日志 | 用户冻结、内容删除、系统配置变更 |
| **数据备份** | 每日自动备份数据库 | 保留7天备份 |
| **HTTPS** | 全站HTTPS | SSL证书配置 |

### 4.3 可用性需求

| 指标 | 要求 | 备注 |
|------|------|------|
| **系统可用性** | ≥99%（V1.0） | 允许计划内维护（凌晨2~4点） |
| **数据持久性** | ≥99.9% | 数据库主从备份 |
| **故障恢复时间** | ≤30分钟 | 自动故障转移+人工介入 |
| **错误处理** | 用户友好错误提示，不暴露技术细节 | 错误码+友好文案 |
| **降级策略** | 内容安全API故障时，进入人工审核模式 | 不影响用户发布，仅延迟展示 |

## 5. 数据模型（核心表结构）

> 详细字段定义见技术架构文档3.3，此处仅补充优化后的关键说明

### 5.1 用户表（users）- 优化后

| 字段名 | 类型 | 约束 | 说明 | 补充 |
|--------|------|------|------|------|
| id | bigint | PK, AI | 用户ID | - |
| phone | varchar(20) | UNIQUE | 手机号 | 用于登录 |
| password_hash | varchar(255) | NOT NULL | 密码哈希 | bcrypt |
| guardian_type | enum | NOT NULL | 守护灵类型 | 'mechanic','elf','astrologer' |
| level | int | DEFAULT 1 | 领地等级 | 1~5 |
| exp | int | DEFAULT 0 | 当前经验值 | - |
| invite_code | varchar(10) | UNIQUE | 个人邀请码 | 6位字母数字 |
| invited_by | bigint | FK | 邀请人ID | 可为NULL |
| territory_coord_x | int | NOT NULL | 领地X坐标 | -1000~+1000 |
| territory_coord_y | int | NOT NULL | 领地Y坐标 | -1000~+1000 |
| signature | varchar(20) | DEFAULT '' | 个性签名 | 需过内容安全 |
| gold_coins | int | DEFAULT 100 | 金币余额 | 0~99999 |
| role | enum | DEFAULT 'player' | 用户角色 | 见2.1 |
| status | enum | DEFAULT 'active' | 账号状态 | 'active','frozen','deleted' |
| created_at | datetime | DEFAULT NOW() | 注册时间 | - |
| updated_at | datetime | AUTO UPDATE | 更新时间 | - |

**补充规则**：
- territory_coord_x和territory_coord_y必须唯一（不可重叠）
- invited_by形成链表结构，可追溯完整邀请链
- 坐标分配算法：邀请人坐标周围±10格内随机，如已被占用则向右偏移+1

### 5.2 邀请记录表（invite_records）- 优化后

| 字段名 | 类型 | 约束 | 说明 | 补充 |
|--------|------|------|------|------|
| id | bigint | PK, AI | 记录ID | - |
| inviter_id | bigint | FK, NOT NULL | 邀请人ID | - |
| invitee_id | bigint | FK, UNIQUE | 被邀请人ID | 一个用户只能被一个邀请人邀请 |
| invited_at | datetime | DEFAULT NOW() | 邀请时间 | - |
| is_active | boolean | DEFAULT false | 是否活跃 | 被邀请人活跃7天后变为true |
| mini_program_generated | boolean | DEFAULT false | 是否已生成小程序 | 防止重复生成 |

**关键规则**：
- is_active=true后才计入邀请人进度（防止刷邀请）
- mini_program_generated标志位防止重复生成小程序

### 5.3 其他核心表

**领地演进记录表（territory_evolution）**：记录每次升级的历史，用于数据分析

**俱乐部表（clubs）**：
- 补充字段：member_count（成员数）、post_count（帖子数）、last_active_at（最后活跃时间）
- 创建条件：用户Lv.2以上

**帖子表（club_posts）**：
- 补充字段：status（'pending','approved','rejected'）
- AI内容安全API返回结果存储在单独的内容审核表（content_audit_logs）

**Agent对话日志表（agent_dialogues）**：
- 补充字段：delivered_at（送达时间）
- 用于分析Agent话术效果，优化V2.0的大模型训练数据

## 7. 技术架构关键点（补充）

### 7.1 OpenIM二改边界（明确版）

| 层级 | 是否修改 | V1.0具体内容 | 技术实现 |
|------|----------|--------------|----------|

| 群组 | ✅ 使用 | 每个俱乐部对应一个OpenIM群组 | 创建俱乐部时自动创建群组 |
