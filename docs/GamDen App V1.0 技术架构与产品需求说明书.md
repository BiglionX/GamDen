# GamDen App V1.0 技术架构与产品需求说明书

**项目名称**：GamDen（游戏巢穴）
**文档版本**：V1.0
**状态**：待评审
**关联文档**：《游客模式体验设计需求文档》、《UI/UX设计规范》


## 一、项目概述

### 1.1 产品定位
GamDen是一个**独立于算法的游戏巢穴社区**——玩家入驻即拥有领地，通过邀请好友扩张版图，在守护灵（Agent）的引导下建立自己的游戏社交圈。

**一句话说清楚**：给“对微信抖音很烦”的游戏玩家一个独处的角落。

### 1.2 V1.0 核心目标
- **验证核心假设**：目标用户（数码粉、游戏玩家）是否愿意为了“领地+守护灵”的玩法留下来并邀请好友。
- **跑通裂变闭环**：从App下载 → 入驻 → 邀请3人 → 生成个人小程序 → 更多下载。
- **建立品牌认知**：让“GamDen = 游戏巢穴”成为第一批用户的共识。

### 1.3 V1.0 功能边界（明确不做啥）

| 做（V1.0） | 不做（V2.0再做） |
|-----------|----------------|
| 领地系统（入驻、升级、元素图演变） | 领地皮肤/装饰商城 |
| 地图（查看邻居、无人区迷雾、野兽踪迹动画） | 手动攻击野兽、组队讨伐 |
| 守护灵（固定话术模板 + 条件触发） | 大模型驱动的个性化Agent |
| 邀请裂变（邀请码、3人解锁个人小程序） | 邀请排行榜、梯度奖励 |
| 俱乐部（按游戏名称建“茶摊”、发帖/回帖） | 俱乐部排行榜、香火值竞赛 |
| 集市（虚拟道具，金币兑换） | 实体周边、盲盒、物流 |
| OpenIM基础版（单聊+群聊+自定义消息） | Agent自动介入聊天、语音房 |
| 游客模式（先逛后注册） | — |
| **展会（游戏推荐）** | **❌ 仅“即将开市”占位图** |


## 二、技术架构

### 2.1 技术选型总览

| 层级 | 选型 | 理由 |
|------|------|------|
| **跨端框架** | UniApp（Vue3 + Vite） | 一套代码编译iOS/Android/小程序/H5 |
| **渲染模式** | UniApp 原生渲染（非WebView） | 性能接近原生，消除跨层桥接消耗 |
| **编程语言** | TypeScript | 类型安全，Cursor对TS支持极好 |
| **状态管理** | Pinia | Vue3官方推荐，轻量易用 |
| **UI组件库** | uview-plus 或 uvui-ts | 基于Vue3 + TS，开箱即用 |
| **即时通讯** | OpenIM（私有化部署 + uni-app插件） | 数据自主可控，支持深度定制 |
| **内容安全** | 腾讯云/阿里云内容安全API | 按量付费，初期成本极低 |
| **后端** | Node.js（NestJS）或 Go | 根据团队技术栈自选 |
| **数据库** | PostgreSQL + Redis | 关系型数据 + 缓存/会话 |

### 2.3 OpenIM 集成方案

根据OpenIM官方文档，UniApp集成OpenIM的标准路径如下：

**核心依赖**：
```bash
npm install openim-uniapp-polyfill @openim/client-sdk
```
- `openim-uniapp-polyfill`：统一封装JSSDK和原生插件，兼容一套代码同时开发APP、H5、小程序
- `@openim/client-sdk`：JSSDK，运行H5和小程序时的必要依赖

**关键集成步骤**：
1. 在DCloud插件市场绑定OpenIM原生插件（manifest.json → App原生插件配置 → 云端插件导入）
2. 在项目中引入：`import IMSDK from 'openim-uniapp-polyfill'`
3. 初始化SDK、登录、绑定事件监听

**二改边界（重要！）** ：

| 层级 | 是否修改 | 说明 |
|------|----------|------|
| OpenIM核心源码 | ❌ 不修改 | 保持原版，方便后续升级 |
| Webhook中间件 | ✅ 新增 | 监听特定消息触发业务逻辑 |
| 自定义消息类型 | ✅ 扩展 | 守护灵通知、领地事件等 |
| 用户体系 | ✅ 对接 | 用户ID与业务后台保持一致 |

### 2.4 环境要求

- **Node.js**：18+ 或 20+
- **HBuilderX**：4.34+ 正式版（推荐，内置编译依赖）
- **OpenIM Server**：v3.8.2+


## 三、功能需求

### 3.1 用户身份与权限（游客 vs 注册用户）

| 功能 | 游客 | 注册用户 |
|------|------|----------|
| 浏览地图（缩放/拖动） | ✅ | ✅ |
| 查看邻居领地 | ✅（坐标模糊化） | ✅（精确坐标） |
| 点击邻居→私聊 | ❌ 引导注册 | ✅ |
| 浏览俱乐部帖子 | ✅（完整阅读） | ✅ |
| 点赞/回复帖子 | ❌ 引导注册 | ✅ |
| 浏览集市道具 | ✅ | ✅ |
| 兑换道具 | ❌ 引导注册 | ✅ |
| 邀请好友 | ❌ 入口隐藏 | ✅ |
| IM聊天 | ❌ 模块不可见 | ✅ |
| 守护灵软引导 | ✅（浏览30秒后触发） | ✅（完整） |

> 游客模式详细设计见独立文档《GamDen游客模式体验设计需求文档》

### 3.2 领地系统

**核心流程**：
1. 用户注册时选择守护灵（机械师/精灵/占星师）→ 系统分配随机坐标 → 获得Lv.1领地
2. 领地随等级提升演变图标（🌱树苗 → 🌿小树林 → 🏡木屋 → 🏘️石屋群 → 🏯小型寨落）
3. 等级通过**签到、发帖、邀请好友**获得经验值提升

**数据模型（关键字段）** ：
```
territories:
  - user_id (FK)
  - coord_x, coord_y (整数坐标)
  - level (1-5)
  - exp (当前经验值)
  - guardian_type (mechanical/elf/astrologer)
  - invited_count (当前邀请人数)
```

### 3.3 地图系统

- **2D网格画布**：以用户坐标为中心，渲染周围±10格的邻居领地
- **无人区迷雾**：未探索格子以深色覆盖，边缘微光动画（为V2.0野兽玩法预留视觉钩子）
- **邻居卡片**：点击弹出半透明卡片，显示玩家昵称、守护灵类型、领地等级、最近常玩游戏标签

### 3.4 守护灵（Agent）V1.0

**V1.0采用固定话术模板 + 条件触发**，暂不接入大模型API：

| 触发事件 | 机械师 | 精灵 | 占星师 |
|---------|--------|------|--------|
| 每日首次登录 | “检测到信号，欢迎回巢。” | “晨光落在你领地上了。” | “星轨显示，今日宜签到。” |
| 领地升级 | “锻造炉温度上升。” | “树又长高了。” | “星力涌动，境界突破。” |
| 邀请成功 | “新坐标已锁定。” | “新芽破土了！” | “命运之线已连接。” |
| 游客浏览30秒 | “巢穴的门是开着的。” | “风在等你做出决定。” | “星象显示，你即将入局。” |

### 3.5 邀请裂变与小程序

**触发条件**：用户邀请好友 ≥ 3人

**自动生成**：
- 系统为该用户生成**个人专属小程序码**和**小程序链接**
- 小程序内容：领地缩略图 + 邀请进度条 + 用户自定义签名
- 所有交互按钮统一引导“下载GamDen App”

**技术实现**：微信小程序与App共用同一套UniApp代码，通过条件编译区分平台

### 3.6 俱乐部（游戏贴吧）

- 每个俱乐部 = 一个OpenIM群组
- 列表页：古风“茶摊”风格，显示游戏名称、成员数、今日新帖数
- 详情页：沿用OpenIM聊天界面框架，替换为GamDen皮肤（暗金主色 + 古风边框气泡）

### 3.7 集市（虚拟道具商城）

- V1.0仅支持**金币兑换虚拟道具**（头像框、领地装饰、升级加速券）
- 无真实支付流程，金币通过签到、发帖、邀请获得
- V2.0再接入微信/支付宝支付 + 实体周边


## 四、非功能需求

### 4.1 性能要求

| 指标 | 目标值 |
|------|--------|
| App首屏加载时间 | < 2秒（原生渲染，无WebView） |
| 地图滑动帧率 | ≥ 60fps |
| 消息收发延迟 | < 500ms |
| 小程序包体积 | < 2MB |

### 4.2 兼容性

| 平台 | 最低版本 |
|------|----------|
| iOS | 13.0+ |
| Android | 5.0（API 21）+ |
| 微信小程序 | 基础库 2.0+ |

### 4.3 安全与合规

- **内容安全**：必须接入腾讯云/阿里云内容安全API（用户签名、帖子内容、私聊消息均需过审）
- **用户隐私**：游客模式仅记录设备ID，不采集个人信息
- **数据存储**：用户数据存储于自有服务器，不依赖第三方云IM（OpenIM私有化部署的意义）

### 4.4 可扩展性

- 领地、守护灵、俱乐部等核心模块均预留V2.0扩展接口
- “展会”模块在V1.0仅做UI占位，代码框架预留，V2.0无缝接入游戏联运

## 七、技术风险与应对

| 风险 | 应对措施 |
|------|----------|
| UniApp x 与 Vue3 兼容性问题 | 使用 HBuilderX 4.34+ 正式版，严格遵循官方模板 |
| OpenIM 多端 API 差异 | 使用 `openim-uniapp-polyfill` 统一封装，必要时用条件编译 |
| 小程序审核被拒（类目问题） | 定位为“工具-信息查询”，避免“社交”标签；V1.0不含支付和游戏内容 |
| 消息推送保活 | OpenIM原生插件直接调用系统推送（APNs/FCM），无需额外开发 |
| 游客→注册转化率低 | 埋点监控转化漏斗，优化守护灵引导话术和触发时机 |
